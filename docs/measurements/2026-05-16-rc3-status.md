# RC3 Status Doc (2026-05-16)

> Snapshot of the `rc3` branch executing the AOT-hardening plan
> ([`docs/milestones/aot-hardening-plan.md`](../milestones/aot-hardening-plan.md)).
> Captures iter progress for an eventual `1.0-rc3` tag decision.

## Headline

RC3 graduates the AOT pipeline along all six hardening axes
without depending on out-of-session work (cs-vm crates.io publish).
Across 10 in-session iters spanning Phases 1, 2, 4, 5, and 6:

- **Distribution prep**: `CsVmDep` enum lets the emitter target
  either path-based (dev tree) or crates.io-version deps. Default
  stays path; flip on once cs-vm is published.
- **Coverage**: multi-block `let` AOTs cleanly via the new
  pre-scan + tier-strict demote pass.
- **Correctness**: demote-pass atomicity bug fixed (the
  spectral-norm "v17 used before defined" surfaced in RC2 is gone).
- **Diagnostics & UX**: Every `UnsupportedInst` error now carries
  a user-meaningful description + suggested workaround.
  `crabscheme aot --explain` surveys lambdas + reports
  AOT-compatibility per entry. `crabscheme aot-doctor` self-
  tests the install end-to-end.
- **Performance**: LTO + codegen-units=1 in the emitted release
  profile; fact-aot binary 633 KB → 584 KB (-7.7%).
- **Hardening**: `--version` on every AOT'd binary embeds the
  crabscheme version that produced it.

## Path to 1.0-rc3 tag

The current rc3 branch is in a tag-able state. Two paths:

**A. Ship now**: Merge rc3 → main, tag `1.0-rc3`. The 10 iters
shipped are strict improvements over RC2; users get better
diagnostics, AOT-binary provenance, smaller binaries, and the
multi-block-let coverage. The deferred iters are all post-RC3
work that don't gate this release.

**B. Wait for crates.io publish**: Phase 1.2-1.4 unblock "release-
installed binary can AOT" which is a bigger user-facing leap
than the diagnostics + perf wins. If shipping RC3 binaries via
the GitHub Releases page is the immediate goal, waiting on
crates.io publish makes sense.

Recommendation: **ship now (A)**. The crates.io publish is
its own release-engineering project; tying rc3 to it would delay
the diagnostics + correctness wins for no good reason. Track
publishing as 1.0-rc4 work.

## Reproduce this state

```bash
git checkout rc3
cargo test --workspace --release
cargo test -p cs-aot                  # 49 tests
crabscheme aot-doctor                 # 8 pipeline checks, all OK
crabscheme aot bench/microbench/scheme/fib.scm --explain
# → 1 lambda, ✓ fib compatible
crabscheme aot bench/microbench/scheme/fib.scm --entry fib --build
./fib-aot/target/release/fib --version
# → compiled by crabscheme (cs-aot 0.0.1) from entry `fib` (NB ABI)
./fib-aot/target/release/fib 35       # → 9227465
```
