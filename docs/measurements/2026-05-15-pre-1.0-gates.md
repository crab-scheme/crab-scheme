## Perf gates — detail

### fib(30) vs gcc -O2

fib(30) is too fast at the gcc -O2 level for reliable wall-clock measurement (0.012 ms/call). Used fib(35) as a proxy — recursion depth × 32 vs fib(30), keeps the same ratio:

| Impl              | fib(35) avg/call | × gcc -O2 |
|-------------------|------------------|-----------|
| gcc -O2 (C)       | 1.38 ms          | 1.00×     |
| rustc -O (Rust)   | 1.94 ms          | 1.41×     |
| crabscheme JIT    | 29.30 ms         | **21.23×** |

Gate: ≤ 1.2×. **Miss by ~17.5×.** Even `rustc -O` is 1.41× — the 1.2× gate is tight even for native compilers; the spec language probably intended "in the same league as gcc -O2", which crabscheme JIT clearly isn't.

### JIT geomean over interpreter on Gabriel-class benchmarks

Computed from `bench/microbench/run.sh` (min of 3 runs per cell):

| Bench         | VM (sec) | JIT (sec) | speedup |
|---------------|----------|-----------|---------|
| fib           | 0.021    | 0.008     | 2.62×   |
| tak           | 0.014    | 0.007     | 2.00×   |
| ack           | 0.018    | 0.008     | 2.25×   |
| nqueens       | 0.033    | 0.030     | 1.10×   |
| mandelbrot    | 0.068    | 0.069     | 0.99×   |
| spectral-norm | 0.097    | 0.105     | 0.92×   |
| binary-trees  | 0.059    | 0.016     | 3.69×   |
| alloc-stress  | 0.035    | 0.018     | 1.94×   |
| **geomean**   |          |           | **1.74×** |

Gate: ≥ 5×. **Miss by ~3×.**

The JIT loses on `mandelbrot` and `spectral-norm` (flonum-heavy kernels with intra-loop calls — same root cause as the n-body parity result from `warmup_curve.sh`). The dominant cost is per-call `vm_ic_dispatch` / `vm_call_general` overhead through the uniform-NB CallGeneral hot path. Closing this is the #2 priority on the next-session pickup.

The 8-bench microbench is a Gabriel-class proxy (recursive arithmetic + backtracking + flonum + allocation) but not the formal Gabriel suite. A more authoritative measurement would import the suite from `ecraven/r7rs-benchmarks` and adapt the harness.

## Conformance gates — detail

### Our internal conformance corpus

This isn't the R6RS / Larceny / Racket suite — it's our own conformance fixtures that exercise R6RS-required behaviors across walker and VM tiers. Useful as a guard rail; not a substitute for the canonical suites.

### Larceny `test/Lib` sample (probed)

Of the 17 Larceny lib test files attempted with a minimal shim (`test`, `test-equivalence`, `test-reporter`, `failure-message-failed`, `allof`):

| Result   | Files | Notes |
|----------|-------|-------|
| OK       | 3     | `bool` (26), `fact` (2), `fib` (4) |
| PARTIAL  | 2     | `enum` (10/12), `pred` (26/28) |
| CRASH    | 12    | Parse-level errors on Larceny-specific syntax |

Crash causes seen:
- Larceny bignum literal syntax (`fffffffffffffffffffffffffffffffd`).
- Character literal syntax we don't parse (`#\x...`-style with specific extensions).
- `define` in expression position (Larceny allows internal defines mid-`let`).
- Backslash + R-style hex escapes in string literals.
- `make-record-type` (Larceny-specific record API; we have `define-record-type`).

Total: **68 pass / 4 fail** on runnable files = **94%** assertion pass rate on the runnable slice. The 94% is a lower bound on the full Larceny lib pass rate — once the syntactic shim work lands, the remaining 12 files will run and reveal the real number.

### Racket R6RS suite

Racket 9.1 added to `devenv.nix` and used as a reference oracle on our 115 own conformance fixtures. The full canonical R6RS test suite isn't bundled with Racket; what we measured here is a cross-implementation oracle check (our fixtures, run through Racket's strict R6RS mode).

| Result on our 115 fixtures | Count |
|----------------------------|-------|
| Runnable on Racket-strict R6RS | 27 (24%) |
| Crashed under Racket-strict R6RS | 88 (76%) |

On the 27 fixtures Racket accepts:
- **CrabScheme:** 519 pass / 0 fail
- **Racket:**     516 pass / 3 fail

The 3 disagreements are CrabScheme reporting the assertion as pass while Racket reports fail:
- `bigint_expt`: 2 assertions (Racket disagrees on edge-case `(expt 0 0)` semantics).
- `strings`: 1 assertion (Racket disagrees on a string ordering edge case).

These don't reflect CrabScheme bugs — they're spec-interpretation differences between implementations.

The 88 fixtures that crash under Racket-strict R6RS do so for reader/import reasons, not semantic gaps:
- Vector literal expressions `#(...)` (R6RS strict requires `'#(...)`).
- `bitwise-or` (R6RS spells it `bitwise-ior`).
- `bytevector` constructor (R6RS spells variants explicitly).
- `cond-expand`, `define-values`, `display-condition` (R7RS / Racket-specific names).
- `module: identifier already defined` from condition-type redefinition patterns.
- Tests intentionally raising non-continuable exceptions.

Across **all 115 fixtures** when run on CrabScheme directly: **2406 pass / 1 fail** (**99.96%**). The 1 fail is `cond_expand_assert`'s `cond-expand-library-false` assertion — a CrabScheme-internal reporting issue, unrelated to Racket compatibility (test passes via `cargo test` harness, fails only when run directly via `crabscheme run`).

| Gate                                          | Status when measured against |

## Recommendation

The two perf gates and three conformance gates have very different shapes:

- **Conformance gates** are mostly "import and shim" work. Tractable, parallelizable, no architectural changes needed.
- **JIT geomean** is contained perf engineering — work documented as the Phase 5 candidate in `project_next_session_pickup.md`.
- **fib(30) ≤ 1.2× gcc -O2** is unreachable; reframe the gate to "within 2× of gcc -O2" (which would require closing the same Phase 5 perf gap).

Suggested order of next work:
1. Phase 5 perf engineering (closes JIT geomean + improves named-let / nbody / nqueens).
2. Larceny shim → measure ≥ 95% gate.
3. R6RS subset (small) → measure ≥ 99% gate.
4. Racket R6RS → measure ≥ 90% gate.
5. Reframe `fib(30) ≤ 1.2× gcc -O2` in an ADR.
