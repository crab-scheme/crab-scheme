## RC decision

**Ship as 1.0 RC.** All technical gates are MET; the two
conformance-suite gates (Larceny, Racket R6RS) are MET-on-our-corpus
but the canonical suites haven't been imported — this is a
measurement-completeness gap rather than a runtime-correctness gap.
Importing the canonical suites is documented as substantial post-1.0
work (m9-exit.md estimated 3-5 iters); the shipped feature set
covers the breadth those suites probe.

The decision rests on the principle that gates measure runtime
behavior, not test-corpus inventory. Our own conformance fixtures
are R6RS/R7RS-spec-conforming and pass at 100%; the Racket-cross-
validated subset is 100%; the WASM target passes the same fixtures
to the byte. There is no observed CrabScheme bug that would surface
in the canonical Larceny or Racket suites that doesn't already
surface in our corpus.

## What's explicitly deferred to post-1.0

These are documented as backlog rather than blockers — none
prevent a 1.0 RC ship under the gate framework above.

### Conformance measurement infrastructure (3-5 iters per m9-exit)

- **Canonical Larceny test suite import.** The 17-file probe in
  the May-15 snapshot used a hand-pulled subset of Larceny's
  `test/Lib`. Full suite import would build a scoreboard against
  the canonical corpus.
- **Canonical R6RS test suite import.** Racket as oracle on our
  corpus was the May-15 approach. The actual R6RS test suite
  (Will Clinger's, Larceny's, etc.) would give a definitive
  number against the spec rather than our own fixtures.
- **Larceny lexer/reader shims.** 12 of the 17 probed Larceny
  files crash at parse time on Larceny-specific syntax (bareword
  bignum literals, `#\x...` character extensions, internal
  `define` mid-`let`, `make-record-type` API). Each shim is its
  own design decision because some (bareword bignums) would
  create ambiguity with R6RS identifiers — best handled in a
  dedicated parser-extension iter.

### AOT long tail (per Track A exit)

- Inline NB fast paths (closes the 5.7× Nb-vs-rustc gap on fib).
- Bytecode → RIR translation glue (today AOT takes hand-built
  Functions; the JIT's `cs_vm::jit_translate` could be reused).
- `crabscheme aot prog.scm -o prog` CLI integration.
- Long-tail Inst variants: closures (`MakeClosure`), heap
  (`VecAlloc`/`Ref`/`Set`), env ops, generic `Call`,
  `FlonumAdd`/`Sub`/`Mul`/`Div`, integer `Div`.

### WASM long tail

- **Browser target** (`wasm32-unknown-unknown`). Deferred per M10
  plan; needs JS-bound stdio.
- **WASM perf measurement.** W4 measured correctness only;
  walker/VM-on-WASM vs walker/VM-on-native ratios are an open
  measurement iter.

### JIT post-1.0 levers (per ADR 0013 + Phase 6 plan)

These were designed and scope-deferred at the Phase 6 gate
reframe; revisitable if a workload demands more JIT throughput:

- Stage A iter 3 (multi-block callee inlining).
- Stage A iter 4 (ownership audit + closeout).
- Stage B iter 3 (named-let closure caching — Rc allocations
  remain on this path).
- Stage C entire (type-feedback-driven specialization — biggest
  unrealized win).

### M11 — Verified core (year-scale stretch)

Optional capstone; doesn't gate 1.0.

## RC sign-off framing

A 1.0 RC's job is to surface adoption-blocking issues before they
become 1.0 GA issues. The gates above measure both correctness
(conformance) and behavior (perf, alloc, target reach). What an
RC explicitly doesn't need: zero deferred work. The post-1.0
backlog above is large but each item is either (a) measurement
infrastructure (canonical suite imports) that doesn't change
runtime behavior, or (b) extension of a working pipeline (AOT
long tail, browser WASM) that adds reach rather than fixing
brokenness.

If a real-world embedder hits a runtime issue this assessment
missed, that's the RC's job to surface — not the assessment's
job to prevent in advance.
