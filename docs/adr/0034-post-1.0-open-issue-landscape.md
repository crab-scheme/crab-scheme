# ADR 0034: Post-1.0 open-issue landscape — #29, #30 part 2, #51

> Status: Accepted (analysis + scoping; no behavior change)
> Date: 2026-05-29
> Authors: crab-scheme contributors

## Context

With the `wasip2-networking` milestone landing on `main` (PR #95), the
in-1.0 surface is closed. Three issues remain open, all explicitly
deferred during the perf/JIT sweep and BEAM v1 exit:

- **#29** — `B7: JIT-invalidation on hot reload (Cranelift safepoints)`
- **#30** — `B3: work-stealing scheduler + auto-yield hook (second
  half)`
- **#51** — `perf/jit: RIR optimization pipeline — unboxing + escape
  analysis`

Each carries a "post-1.0 per the BEAM v1 exit report" or "deferred —
poor risk/reward right now" annotation. This ADR consolidates *why* the
three are deferred, what specific architectural / engineering walls
gate each, and what the concrete next step is when those walls move.
The point is to make sure a future contributor (or future us) doesn't
restart the analysis from zero.

ADR 0032 already did this for #30 part 2 in detail; this ADR repeats
its conclusions in the context of the wider deferral landscape and adds
the #29 / #51 analyses that didn't have their own ADR.

## The walls

### Shared wall — `!Send` `Rc`-everywhere `Value`

ADR 0032 named this for #30 part 2: an actor's Scheme heap is
`Rc<…>`-pervasive (`Value: !Send`) and pinned to its thread. The same
wall surfaces for **#29**:

`cs-runtime::builtins::beam::to_sendable_in` *rejects* `Value::Procedure`
(see the `Err("to_sendable: procedures cannot cross actor boundaries")`
arm). `load-module!` runs every export value through `to_sendable_in`
before handing it to `cs-hotreload::VersionRegistry::load`, so
**procedures cannot be stored in the version registry today**.

That is, the BEAM-style "hot-reload a function, callers keep running
old bytecode, JIT body is stale" scenario in #29's issue body **cannot
occur with the current implementation**. Module exports are
data-only — `(load-module! 'foo '(("init-state" . 0) ("schema-version"
. 1)))` works; `(load-module! 'foo (list (cons "fn" (lambda () 1))))`
errors at the boundary. The `beam_counter_migration` E2E in
`crates/cs-runtime/tests/` exercises this data-only contract end-to-end.

Pretty-printing #29 onto this finding:

| Aspect | State on `main` (2026-05-29) |
|---|---|
| Two-version dispatch (`old` / `current`) | ✅ shipped (B6) |
| Lookup primops (`lookup-code`, `lookup-code-old`) | ✅ shipped |
| Purge primops (`code-soft-purge!`, `code-purge!`) | ✅ shipped |
| `code-change` state-migration callback | ✅ data-only flavor proven by `beam_counter_migration` |
| **Procedures as module exports** | ❌ *blocked* on `Send` heaps / heap-rehydration |
| **JIT-body invalidation on reload** | N/A — depends on the prior row |
| Cranelift safepoints (interrupt in-flight JIT code) | Not started; only relevant after the prior row |

So #29's "proper fix needs Cranelift safepoints" framing is correct
**eventually**, but the prerequisite is the same Send-heaps work that
gates #30's iter-2b. Until module exports can hold a procedure value
that survives an `Arc<dyn Any + Send + Sync>` round-trip, there is no
JIT body for the registry to invalidate.

Two plausible paths past this wall:

- **Isolated heaps for actors** — `Value: Send` via a per-actor heap.
  Same project as ADR 0032's iter-2b. Large GC / value-representation
  rework.
- **Per-actor rehydration** — module exports stored as a *code blob*
  (bytecode + sym table mapping, both already `Send` since they're
  `Vec<u8>` / `String`); each actor that resolves the export
  reconstructs a closure in *its own* heap. Smaller than full Send
  heaps but still substantial; closure environments at define-time
  would need to be sendable (which today they often aren't —
  cross-actor `Rc<Env>` is the same wall in a different shape).

ADR 0032 already accepts iter-2b's deferral; #29 inherits that
deferral verbatim. Neither path is in 1.0 scope.

### #30 part 2 — covered in ADR 0032

ADR 0032 captures the work-stealing analysis: VM-level reduction
preemption is already in place; JIT reduction-tick preemption shipped
in ADR 0031 (#30 iter-1). The Part 2 second half — automatic yield
without `(yield)` *plus* true work-stealing — sits behind the same
`!Send` heap wall as #29. ADR 0032's verdict (ride tokio, then add
`LocalSet` if needed, but only after Send heaps) is the canonical
breakdown; this ADR just notes the shared root cause.

The thread-per-actor model (~4096-actor ceiling via tokio's
`max_blocking_threads`) is accepted for 1.0 and documented as a known
limit (ADR 0032 §Consequences).

### #51 — different wall: JIT correctness regression + zero `with-region` usage

#51's escape-analysis MVP shipped in PR #53; the safety-critical
`#51a` UAF was fixed in PR #54; the JIT loopvar-capture bug found in
the same sweep was fixed in PR #83 (ADR 0030). The remaining
sub-issues — `#51b` (let-temp promotion) and `#51c` (closure escape
analysis) — are deferred for different reasons than #29/#30:

- **Risk/reward is poor.** The "obvious" fix — running
  `demote_env_to_ssa_all_blocks` on the JIT path so let-bound conses
  enter SSA where `escape-to-region` can promote them — regressed
  10+ `jit_differential` tests on the JIT builtin surface (alist /
  bytevector / bitwise lookups left un-rewritten in blocks 1+ →
  use-before-def). The demote is only proven on the narrower AOT
  corpus.

- **Real-world benefit is gated on `with-region` usage that doesn't
  exist.** The promotion only helps code inside an explicit
  `(with-region …)` scope. Standard benchmarks and 1.0 user code
  don't use it. Promoting let-temps that nobody allocates is
  optimization without an observed workload.

- **The "scoped next step" is concrete but still risky.** Broaden
  the demote's block-1+ operand rewrite from the inliner's
  `is_inline_supported` predicate to the strictly-larger
  `cs_rir::inline::value_walker_covers` set (Cons / Car / Cdr /
  predicates), validate against the full `diff_jit_*` suite, *then*
  promote. The hazard is the same as the rejected variant: the demote
  on JIT bodies hits inst variants outside the walker's coverage
  (Bytevector / Bitwise / String ops) and panics from
  `for_each_value_in_inst`'s `unreachable!()` arm. The full operand-
  walker would need extending in lockstep, which is mechanical but
  large.

- **#51c (closure escape) inherits #51b's gating.** Closures capture
  via env (not explicit operands); a non-escaping-closure analysis
  needs the env/region model maturity that #51b is itself waiting on.

So #51 is *not* blocked on the `!Send` heap wall — it's blocked on
JIT-path demote-pass coverage gaps + on `with-region` actually being
used by enough code to justify the engineering. Both are tractable;
neither is in 1.0 scope. The full prior analysis lives in the issue
#51 comment dated 2026-05-21.

## Why no in-1.0 fix landed today

Each of the three has a concrete, code-level next step in the
literature (in this ADR + ADR 0032 + the #51 issue comment). None of
them is a 1.0 deliverable, for distinct reasons:

| Issue | Wall | Tractable now? | Value if delivered today |
|---|---|---|---|
| **#29** | `!Send` `Rc`-everywhere `Value` → procedures cannot reach the version registry | No (large Send-heaps / rehydration project) | Zero — no current path exists to trigger the bug |
| **#30 part 2** | Same `!Send` heap wall blocks free migration; iter-2a (`LocalSet`s) is multi-week | iter-2a possible but expensive; iter-2b not | Medium — removes 4096-actor ceiling |
| **#51 (b + c)** | JIT-path demote coverage + zero `with-region` usage | Risky (last attempt regressed 10+ tests) | Low — gated on usage that doesn't exist |

The 1.0 RC is shipped (rc7 tag, all milestones tagged M0–M10). These
three issues track meaningful future work; closing them today would
either be premature (no actual fix), or destructive (force a fix past
its own documented risk gate). Leaving them open with this ADR linked
is the honest accounting.

## Decision

1. **Keep #29, #30, #51 open** as post-1.0 work.
2. **Link this ADR from each issue** so the next contributor finds the
   architectural map without re-deriving it.
3. **No code changes in this ADR** — design capture only. The small
   adjacent improvement that ships alongside (a clearer
   `to_sendable_in` error message on procedures + an integration test
   pinning the current data-only contract) is mechanical, not
   architectural; it just makes the wall visible at the point a
   contributor first runs into it.

## Concrete next steps when the walls move

Recording the "if you have an afternoon" prompts so they're not lost:

- **#29**, smallest useful step **once Send/rehydration lands**: walk
  the OLD-version exports on `code-purge!`, mark each VmClosure's
  `LambdaProfile.purged = true`, clear `jit_ptr` so the JIT path
  falls back to the walker (which checks the flag). Cranelift
  safepoints are needed only to interrupt code currently *in* a JIT
  body — and purge typically happens between mailbox messages where
  the JIT body has already returned.

- **#30 iter-2a** (LocalSets, no migration): convert
  `spawn_sync_body_on_task` to a `tokio::task::LocalSet`-per-worker
  model so multiple actors share a worker via cooperative yields,
  removing the 1:1 actor:thread mapping. Stay on `!Send` heaps —
  this only multiplexes them, doesn't migrate. ADR 0032 §"Tokio vs.
  custom" already enumerates the trade-offs.

- **#51b**: extend `for_each_value_in_inst` + `value_walker_covers`
  in lockstep with every JIT-emitted Inst variant (Bytevector /
  Bitwise / String / Hashtable ops, …), then enable
  `demote_env_to_ssa_all_blocks` on the JIT path. Validate against
  the full `jit_differential` suite. Only justified by a workload
  using `(with-region …)`.

## Consequences

- No behavior change.
- Documentation eliminates re-derivation cost for the next
  contributor.
- Each open issue gets a clear "what's blocked, what unblocks it,
  what to ship when it does" pointer.
