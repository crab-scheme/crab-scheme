# Optimizer plugins

CrabScheme has a pluggable optimizer that operates on the `cs_rir`
intermediate representation shared between the JIT and AOT tiers. You
can:

- Activate one of the three built-in passes from Scheme code, including
  inside a `#!lang` library so a file opts itself in.
- Write your own pass in Rust against the `cs_opt::Pass` trait, ship it
  as a Rust crate, and have user Scheme code activate it by name.

Same passes run for JIT and AOT — write once, both tiers benefit. The
full design is in [ADR 0014](../adr/0014-optimizer-plugins.md).

## The three built-in passes

| Name | Effect |
|---|---|
| `constant-fold` | Folds `(+ 2 3) → 5`, `(* x 1) → x`, etc. at the IR level. Runs after expansion + macro hygiene, so it sees the post-translation shape. |
| `dead-block-elim` | Drops unreachable basic blocks (typically generated by macro-expanded `cond` / `case`). |
| `inst-stats` | Diagnostic — counts each `cs_rir::Inst` variant. Useful for measuring how much an optimization changed the IR shape. |

Built-ins are registered with the pass registry at `Runtime::new()` —
they're always available by name, just opt-in.

## Activating passes from Scheme

Three Scheme-level builtins control the active-pass list (process-wide,
applied at every JIT/AOT compilation):

```scheme
(install-optimizer-pass! 'constant-fold)         ; activate by name
(install-optimizer-pass! 'dead-block-elim)
(installed-optimizer-passes)                     ; ⇒ (constant-fold dead-block-elim)
(remove-optimizer-pass! 'dead-block-elim)
(installed-optimizer-passes)                     ; ⇒ (constant-fold)
```

`install!` is idempotent (re-installing a name is a no-op). Removing a
name not in the list is also a no-op. Order is install-order — passes
run in the order they were installed.

## Activating passes per file via `#!lang`

ADR 0014's "language as policy" idea: a `#!lang` library installs the
passes its language wants. The minimal example ships at
[`lib/lang/opt-fold.scm`](../../lib/lang/opt-fold.scm):

```scheme
; lib/lang/opt-fold.scm
(library (lang opt-fold)
  (export)
  (import (rnrs))
  (install-optimizer-pass! 'constant-fold))
```

A user file then opts in by header:

```scheme
#!lang opt-fold
(define (compute) (+ 2 3))    ; folded to 5 in the JIT/AOT IR
```

> **Scoping caveat**: the current MVP scopes passes process-wide. A
> file's `#!lang` installs but does not auto-remove on file end. The
> roadmap (R6RS++ Phase 2E) moves `(active-passes)` to a Scheme
> `parameter` so `parameterize` gives true file-scope. Until that
> lands, pair a `#!lang` install with an explicit
> `(remove-optimizer-pass! 'X)` at end-of-file if you need strict
> scope.

## Writing your own pass in Rust

The reference plugin is the `cs-opt-example` crate
(`crates/cs-opt-example/src/lib.rs`). Five-step recipe:

### 1. Add dependencies

```toml
# Cargo.toml
[dependencies]
cs-opt = { workspace = true }
cs-rir = { workspace = true }
```

### 2. Implement the `Pass` trait

```rust
use std::sync::Arc;
use cs_opt::{Bucket, Pass, PassContext, PassRegistry, RegisterError};
use cs_rir::Function;

pub struct MyFastPath;

impl Pass for MyFastPath {
    // Must match [a-z][a-z0-9-]* — that's how Scheme refers to the pass.
    fn name(&self) -> &'static str {
        "my-fast-path"
    }

    // Which optimization bucket — `Folding`, `DeadCode`, `Diagnostic`, ...
    fn bucket(&self) -> Bucket {
        Bucket::Folding
    }

    // The transformation. Mutate `func` in place; record diagnostic info
    // into `ctx.stats` if useful.
    fn run(&self, func: &mut Function, ctx: &mut PassContext) {
        // ... your optimization ...
    }
}
```

### 3. Expose an `install()` function

```rust
pub fn install() -> Result<(), RegisterError> {
    let mut registry = PassRegistry::global()
        .lock()
        .map_err(|_| RegisterError::PoisonedRegistry)?;
    registry.register(Arc::new(MyFastPath))
}
```

### 4. The embedder calls `install()` at startup

In a binary that hosts a `cs_runtime::Runtime`:

```rust
fn main() {
    my_plugin::install().expect("plugin registration");

    let mut rt = cs_runtime::Runtime::new();
    // ... user Scheme code now sees the pass by name ...
}
```

### 5. User Scheme code activates it

```scheme
(install-optimizer-pass! 'my-fast-path)
; ... subsequent JIT/AOT compilations run my-fast-path ...
```

## What `PassContext` gives you

```rust
pub struct PassContext {
    pub bucket: Bucket,         // which bucket the pass is in
    pub stats: PassStats,       // record diagnostic counters here
    pub syms: SymbolTable,      // pass-local symbol table  [known stub — see below]
    pub typer_hints: Option<…>, // per-Value type hints       [known stub — see below]
}
```

Two follow-up gaps (the exit report calls these out):

1. **`ctx.syms` is currently constructed empty for every pass
   invocation.** A pass that needs symbol-name introspection sees
   `syms.len() == 0`. Threading the runtime's real `SymbolTable` is
   the obvious fix, deferred until a pass actually needs names.
2. **`ctx.typer_hints` is currently always `None`.** When the typer's
   per-Value type inference is wired through to optimizer passes, this
   exposes that map. Until then, do your own type detection within
   the pass.

If you're writing a pass that needs either, file an issue describing
the use case — both gaps have known fixes; they're just unbuilt
because nothing has needed them yet.

## Inspecting pass effects

The `inst-stats` built-in records per-`Inst` counts into
`ctx.stats.inst_counts`. You can chain it after your pass to see what
shape the IR has after transformation:

```scheme
(install-optimizer-pass! 'my-fast-path)
(install-optimizer-pass! 'inst-stats)
; ... run code; inst-stats fires after your pass ...
```

For lower-level inspection during pass development, look at the unit
tests in `crates/cs-opt/tests/` — they show the typical
`fold_pass_emits_correct_shape` shape: build a `cs_rir::Function`
by hand, run the pass, assert on the resulting `blocks` and `insts`.
